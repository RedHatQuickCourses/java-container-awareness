When managing Java applications in Red Hat OpenShift or Kubernetes, it is critical to understand whether your diagnostic tools perceive the underlying host node or the isolated container environment. Misinterpreting these outputs often leads to "out of memory" kills or unexpected CPU throttling, because your application may be configured based on node-level visibility rather than container-level constraints.

To effectively troubleshoot Java in containers, you must categorize your diagnostic tools based on their ability to parse cgroups, or control groups.

Let’s start by looking at non-container-aware tools. These report information regarding the physical or virtual host—the OpenShift node itself. If you rely on these tools within a pod, you will get a false sense of resource availability. For instance, the command `lscpu` will report the total CPU architecture and count of the physical worker node, not the specific CPU quota assigned to your pod. Similarly, `nproc` returns the number of processors available to the host operating system, completely ignoring the Kubernetes CPU limits defined in your pod specification. Finally, the `free -m` command displays the total RAM available to the node. If your pod has a 512-megabyte limit but the node has 64 gigabytes, this command will inaccurately suggest you have the full 64 gigabytes available for your Java heap.

In contrast, container-aware tools interface with the Linux cgroups subsystem to identify the specific constraints imposed by the orchestrator. These tools are the true source of truth for your application environment. 

You can use flags like `-XX:+PrintFlagsFinal` or `-XshowSettings:system` to instruct the Java Virtual Machine to report the memory and CPU settings it has calculated based on container limits. These are excellent for verifying if the JVM has correctly identified the container memory boundary. Another powerful option is the `VM.info` command via `jcmd`, which provides a snapshot of the JVM, including internal settings that reflect whether container-aware logic was successfully applied. Additionally, if you are using Red Hat-supported images, the `run-java.sh` script is designed to evaluate the container environment and automatically set optimal Java heap sizes before the JVM even starts.

To develop a solid diagnostic strategy, you should compare the output of host-level tools against JVM-internal tools. While `free -m` and `nproc` are useful for verifying the underlying node capacity, they should be ignored when you are calculating thread pools or garbage collection threads. Instead, prioritize `java -XshowSettings:system` to confirm the JVM sees your pod's memory limits, and use `jcmd` with the `VM.info` command for deep runtime inspection of detected resource limits.

If your application is crashing with an "OOMKiller" signal, follow this diagnostic workflow. First, check JVM visibility by running `java -XshowSettings:system` inside the running container. Second, compare the `MaxHeapSize` and `TotalPhysicalMemory` in that output against your OpenShift deployment resource requests and limits. Finally, review your pod startup logs. If the logs show the JVM is using a default—such as 25 percent of physical memory—rather than a percentage of the container limit, the JVM is likely not properly detecting the cgroup constraints.

A useful tip to keep in mind: If your logs indicate that your application is defaulting to 25 percent of the host node's memory rather than your container's limit, verify that you are using a supported OpenJDK version—specifically 8u191 or higher—and that the image is properly configured to handle cgroups.

In this session, we will explore how to determine CPU availability within containers, specifically when running Java applications.

In traditional environments, the Java Virtual Machine, or JVM, determines the available CPU count by querying the host operating system. However, in containerized platforms like Red Hat OpenShift or Kubernetes, a container is typically restricted to only a portion of the host's total compute resources.

If the JVM is not container-aware, it may incorrectly identify the total CPU count of the underlying host rather than the container's specific limit. This mismatch is important because it can lead to inefficient thread pool sizing—for example, within `ForkJoinPool` or when managing parallel garbage collection threads—which ultimately causes contention and degrades application performance.

Modern JDKs, specifically Java 8 update 191 and all subsequent versions, are designed to be container-aware. They use cgroups, or Control Groups, to accurately determine CPU availability by reading the cgroup filesystem to calculate what is known as the `ActiveProcessorCount`.

The JVM calculates this count by inspecting two primary metrics from the cgroup controller. First, it looks at CPU shares, which represent the relative weight of the container. If the host has many cores, the JVM uses these shares to influence its internal parallelism. Second, it looks at CPU quota and period, which define a hard limit. For instance, if the quota is 200,000 and the period is 100,000, the container is effectively limited to two cores. The JVM determines the number of processors to use by taking the minimum of the host's physical processors and these calculated container limits.

To diagnose how your Java application perceives its environment, it is helpful to distinguish between tools that provide a "Host" view and those that provide a "Container" view. Tools like `lscpu` or `nproc` are not container-aware; they return the total CPU count of the underlying OpenShift node, which is often misleading for capacity planning. In contrast, JVM-specific tools like `VM.info` or the flag `-XX:+PrintFlagsFinal` are container-aware and will report the `active_processor_count` as the JVM actually perceives it.

To verify this in your own environment, you can use the `jcmd` utility. First, access your pod terminal by running the command `oc rsh <pod-name>`. Once you are inside the pod, run `jcmd <pid> VM.info` to view the virtual machine information. When you review the output, look for the container section. A container-aware JVM will explicitly list the `active_processor_count`, allowing you to confirm that the JVM correctly recognizes your resource limits.

When configuring your environment, keep these best practices in mind. Always explicitly define `resources.limits.cpu` in your OpenShift deployment YAML to ensure that cgroup values are populated correctly. Additionally, avoid oversizing; if you set a high CPU limit for a single-threaded application, the JVM might spawn excessive garbage collection threads, leading to unnecessary context-switching overhead. Finally, always monitor the `active_processor_count`. If it does not match your expectations, ensure you are running a supported OpenJDK version. If you are forced to use an older, non-aware version of Java, you must manually set the `-XX:ActiveProcessorCount` flag to match your container's limit.

Let’s wrap up with a quick hands-on exercise. First, identify your Java process ID by running the `jps` command inside your container. Next, query the VM metadata by running `jcmd <pid> VM.info | grep active_processor_count`. Finally, compare this result with the output of the `nproc` command in that same terminal. If `nproc` returns the host's total CPU count—such as 64—but your `active_processor_count` returns your correct container limit—such as 2—you can be confident that your application is correctly container-aware.

When deploying Java applications in containerized environments like Red Hat OpenShift or Kubernetes, it is essential to understand that traditional Linux diagnostic tools often fall short. Tools that are not "container-aware" report the physical or virtual resources of the entire host node, rather than the specific constraints applied to your pod. In this lesson, we will explore the difference between container-aware and node-aware outputs and how to use them effectively for troubleshooting.

Let’s start by looking at the visibility gap. If you use standard Linux utilities such as `lscpu`, `nproc`, or `free -m` inside a container, you are likely seeing the resources of the underlying host node. Relying on these values for capacity planning can lead to serious issues, such as "Out of Memory" kills or CPU throttling, because the JVM might attempt to allocate resources that exceed your container's cgroup limits. To monitor your Java application accurately, you must utilize tools and flags that interface directly with the cgroups filesystem.

The JVM provides built-in mechanisms to report exactly what it sees regarding these container limits. One of the most critical flags for initial diagnosis is `-XshowSettings:system`. When used alongside `-XX:+UnlockDiagnosticVMOptions`, it displays the system information the JVM has detected, including cgroup-enforced memory limits and the active processor count. Additionally, you can use the `jcmd` utility with the `VM.info` command, which provides a comprehensive dump of your current JVM configuration, including those container-specific constraints.

When you run a container-aware command, you need to know what to look for. Focus on three key indicators. First, check the `active_processor_count`, which represents the number of CPUs available to the JVM based on your Kubernetes or OpenShift limits. Second, verify the `memory_limit_in_bytes`. If this value does not match your expected resource limit, the JVM is not correctly detecting the container boundary. Finally, pay attention to `cpu_quota` and `cpu_period`. These values dictate CPU throttling behavior; a quota of negative one suggests no limit is set, while specific positive values indicate restricted processing time.

Now, let’s move to a hands-on activity where you will verify your container limits. Before you begin, ensure you have a running Java pod in your OpenShift or Kubernetes environment and that you have access to `oc rsh` or `kubectl exec`.

First, let’s compare node versus container visibility. Log into your running Java pod by running the command `oc rsh` followed by your `<pod-name>`. Next, run a non-container-aware command like `free -m` or `nproc`. You will notice that the output reflects the total RAM and CPUs of the entire worker node, not your specific container. Now, run the container-aware command: `java -XX:+UnlockDiagnosticVMOptions -XshowSettings:system -version`.

Second, analyze the JVM output. Look for the container information section. You should see entries for `container_type`, `cpu_cpuset_cpus`, `active_processor_count`, and `memory_limit_in_bytes`. Take a moment to verify that the `memory_limit_in_bytes` aligns with the memory limit defined in your pod’s YAML configuration. If these numbers differ, check your JVM version; ensure you are using version 8u191 or later, as older versions lack proper cgroup awareness.

Finally, for real-time analysis, use the `jcmd` utility. While your application is running, execute `jcmd <pid> VM.info`. Search the output for the cgroup section. This provides a real-time snapshot of metrics, including `memory_usage_in_bytes`. You can use this data to determine if your heap size is appropriately configured to leave enough headroom for off-heap memory. 

By utilizing these container-aware diagnostic tools, you can ensure your Java applications are correctly bounded and effectively monitored in your containerized environment.
